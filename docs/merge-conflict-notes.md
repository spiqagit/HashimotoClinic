# マージがうまくいかなかった原因のまとめ

## 今回のマージでコンフリクトしたファイル

- **functions.php**
- **inc/search/search-post.php**

---

## 想定される原因

### 1. 同じ箇所を両方のブランチで編集していた

| ファイル | 衝突しやすい箇所 | 内容 |
|----------|------------------|------|
| **functions.php** | 「投稿関連」セクション（`post_link`・リライトルール・スラッグメタボックス） | **main** と **取り込み元ブランチ** の両方で、投稿のパーマリンク・スラッグ周りを触っていたため、同じ行の前後に追加や修正が入り、Git が自動で結合できなかった。 |
| **inc/search/search-post.php** | ブログ一覧の見出しタグ（`h3` ⇔ `h2`） | 一方のブランチで `h3`→`h2` に変更し、もう一方では `h3` のままだった（または別の修正が同じブロックにあった）ため、同じ行で競合した。 |

→ **「同じファイルの同じ付近」を別ブランチで直すとコンフリクトしやすい。**

---

### 2. main と dev（または作業ブランチ）の役割分担がはっきりしていない

- **main**: スラッグ編集・投稿表示リンク修正（Cursor で実施）
- **取り込み元（c2a167e）**: スラッグメタボックス＋リライト優先度＋search-post の見出し変更 など

どちらのブランチで「正」とするかが分かれておらず、**両方で functions.php と search-post.php を編集していた**ため、マージ時に「どちらの変更を残すか」の衝突が起きた。

---

### 3. 長期間 main と dev を乖離させていた

- `main` にだけ入れた変更と、dev 側で進めていた変更が、**同じテーマの同じファイル**に重なっていた。
- 定期的に dev → main のマージをしていても、**同じ機能（投稿スラッグ・パーマリンク）を両方で触ると**、マージのたびにコンフリクトしやすい。

---

### 4. コンフリクト解消後の「マージ完了」を忘れがち

- コンフリクトをエディタで解消したあと、**`git add` だけして `git commit` していない**状態が続くと、「まだマージの途中」のままになる。
- メッセージに出ていた **「All conflicts fixed but you are still merging」** は、「衝突は解消したが、マージを完了するコミットをまだしていない」という意味。

→ コンフリクト解消後は必ず **`git add` → `git commit`** でマージを完了させる必要がある。

---

## 今後のマージをスムーズにするためのポイント

1. **同じ機能は一つのブランチでまとめて作業する**  
   例: 投稿スラッグ・パーマリンクは main か dev のどちらか一方でだけ変更する。

2. **dev をこまめに main にマージする**  
   長く分離すると、同じファイルを両方で触る機会が増え、コンフリクトが増える。

3. **コンフリクトが出たら「どちらの意図を採用するか」を決めてから解消する**  
   両方の「良いとこ取り」にすると、意図が混ざってバグの元になりやすい。  
   「main を正とする」「dev を正とする」のどちらかを決めてから編集する。

4. **マージは「衝突解消 → add → commit」まで一気にやる**  
   コンフリクト解消後は、忘れずに `git commit` でマージコミットまで完了させる。
